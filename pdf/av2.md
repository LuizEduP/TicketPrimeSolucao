### PARTE 2: ENTREGA DA AV2 (Qualidade, Arquitetura e Operação)

**O que você vai programar na AV2**
Na AV1 você provou que sua API compila, conecta ao banco e faz operações básicas (CRUD). Agora, na AV2, você vai evoluir o sistema com complexidade real de negócio e vai produzir o SDD (Software Design Document) — o conjunto de artefatos que prova que o sistema foi projetado com Engenharia de Software, e não apenas "codado".

**Requisitos de Código (agnósticos de domínio)**
Com base nas Histórias de Usuário do seu backlog, implemente na sua Minimal API:
1. Pelo menos 2 novos endpoints (ou evoluções significativas de endpoints existentes) que implementem regras de negócio do seu domínio — não podem ser apenas CRUD simples de 1 tabela.
2. Pelo menos 1 endpoint deve envolver JOIN: Uma consulta SQL com INNER JOIN OU LEFT JOIN que retorne dados relacionados de duas ou mais tabelas.
3. Pelo menos 1 endpoint deve aplicar múltiplas validações de negócio: Antes de executar o INSERT/UPDATE no banco, o código C# deve verificar pelo menos 3 condições de negócio. Se qualquer validação falhar, a API deve retornar 400 Bad Request com mensagem específica.
4. Todas as queries continuam parametrizadas: Assim como na AV1, nenhuma concatenação ou interpolação de string em comandos SQL. Use exclusivamente `@Parametro` no Dapper.

**O que NÃO é avaliado na AV2**
A AV2 não avalia se o domínio do seu sistema é "criativo" ou "complexo". Um sistema de agenda de consultas, um controle de tarefas, um marketplace ou um catálogo de filmes — todos têm o mesmo peso. O que importa é a qualidade da engenharia aplicada sobre ele, materializada nos 20 artefatos abaixo.

**Tabela de Avaliação AV2 (10 Pontos = 20 itens × 0,5)**
Os 20 itens abaixo compõem o SDD do seu projeto. Cada item é verificado automaticamente pela presença de padrões específicos nos arquivos do repositório. Leia com atenção: o que está entre é o literal que a IA vai buscar.
